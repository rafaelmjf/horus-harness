# Product audit — 2026-07-20 (horus 0.0.73) — inward alignment analysis

**What this document is.** A periodic inward look at Horus itself: does what the
code actually delivers still match the product vision? It reports analysis and
suggestions only and **decides nothing** — verdicts on the vision's parts belong
to a later convergence step (paired with an outward market scan), new card
proposals to the scoping step, and every archive/improve/ready decision to the
final backlog-refinement step.

**Window:** v0.0.58 → v0.0.73 (2026-07-16 → 2026-07-20; ~100 merged PRs, 105
shipped work items in the archive). **Evidence:** git history and the shipped
ledger; a reference-grep of every CLI command across the places that integrate
them (instruction blocks, hooks, terminal UI, dashboard, skills, scripts);
machine-local usage state (3 dispatch authorizations, a 15-entry schedule
ledger with 9 real/test unattended runs, 97 recorded model-performance datums,
a live Telegram alert channel); the 2026-07-20 architecture-boundary study;
Claude Code / Codex release notes; the previous audit receipt (2026-07-16).

## 1. The product, in plain terms

Horus is a small tool that lives inside a project's git repository and acts as
a **product owner's memory and working rituals for AI coding agents** (Claude
Code, Codex). Any fresh agent session can read the repo and know what the
project is, what was decided, what shipped, and what to do next — and the owner
can see and steer all their projects from a terminal, browser, or phone.

A 2026-07-20 architecture study made the delivered shape explicit. Horus is
three layers, and the layering is now verified in code rather than assumed:

1. **Utility substrate** — the plumbing that works with any workflow: launching
   agent sessions under multiple isolated provider accounts, terminal sessions
   that survive and can be re-attached from desktop or phone, scheduling
   unattended runs, bounded pre-authorizations for those runs, a two-way
   notification bridge, honest usage/rate-limit readings, and a hosted
   dashboard. This layer never reads the project-planning files.
2. **Continuity contract** — the product's actual API: a handful of
   machine-read fields. A *session tier* (five frontmatter fields in the
   project's PRD file) is all that resume and the cockpit consume; a *dispatch
   tier* (a per-work-item card schema with readiness labels) is what unattended
   execution consumes. One production project (fabric) runs on the session
   tier alone, proving the tiers are separable.
3. **Workflow policy** — the opinionated product-owner rituals (research →
   vision → roadmap → scope → refine) shipped as agent skills, plus the prose
   shape of the planning files. Optional by construction: nothing in the code
   breaks if they are never invoked.

Two structural findings from this window sharpen the thesis. First, the product
improved *by subtraction*: a "session mode" axis and a continuity-granularity
setting shipped and were deleted within days once they proved to be prose
pretending to be control — authority now lives in the agent CLI's enforced
permission posture, state in committed files. Second, the unattended-execution
loop became real end-to-end (pre-authorize → scheduled run → independent
verification → merge or escalate), quietly extending Horus from "memory +
rituals" to "memory + rituals + a bounded execution loop under owner
authority" — still deliberately below full orchestration, which the vision
excludes. The owner's frame: this repo is the *engine*; the intended *games*
are data/BI projects with fewer software-engineering formalities, which biases
the roadmap toward substrate + contract over deeper workflow ceremony.

## 4. Where each facet stands — detail

### Continuity core

*DoD: a fresh agent session resumes the exact next step from durable state
alone, fetch-first, across machines.* This is the proven spine, and this very
session is its evidence: it resumed from the previous session's authored
handoff, verified the remote state, and delivered within minutes. The window's
main work was convergent **deletion**: a "session mode" axis (prose telling the
model how much process to perform) and a per-project granularity setting were
removed in favor of one rule — git preserves every delivery between
boundaries; planning prose is consolidated once at a real boundary — with
authority moved into the launch permission posture that the agent CLI itself
enforces. The closure ritual's one sanctioned direct-push exemption is now
enforced by code rather than trusted. Remaining gap: the machine-read contract
(section 1, layer 2) is implicit in two parser modules; an already-drafted
work item would declare it explicitly and close the facet.

### Dashboard / cockpit

*DoD: owner sees fleet state and launches/resumes any project from web or
phone, no terminal command.* Functionally met — but by the terminal UI reached
over SSH (including from a phone SSH app), not by the web dashboard the DoD
imagined; the hosted web app is a read-mostly companion behind an access
gate. This window added a consolidated one-form launch, a per-card field
picker, visible outcomes for scheduled runs, and minimal phone push. The
honest convergence question: did TUI-over-SSH *win*, making web-launch a
non-goal? Separately, five commands from the pre-terminal-UI era survive with
zero references and zero observed usage across two audits: an always-on-top
desktop companion (`app`/`mascot`), VS Code task helpers
(`vscode-task`/`vscode-open`), a window-raiser (`focus`), a brainstorm helper
(`brainstorm`), and a token-overhead estimator (`overhead`).

### Accounts & isolation

*DoD: every account isolated by default; cross-account corruption impossible;
usage visible per account.* Converged on every clause. Each provider account
gets its own isolated configuration directory, provisioned automatically and
checked by the doctor command; the one real corruption incident's root cause
dissolved with isolation, after which the launch-refusal guard was *relaxed*
on evidence — the mature direction (guards earn their place from field
failures, and stand down the same way). Usage is read from the surface the
agent app itself pushes rather than a polled endpoint that throttles, carries
a source and an age, and account names resolve exactly or are refused.

### Delegation calibration

*DoD: agent picks execute-vs-dispatch + a model tier from live measured data,
owner-gated, honest cost — never auto-routing.* All mechanics exist: 97
recorded run datums, a capability ladder, an explicit consent envelope for
every worker launch, honest completion accounting. But the facet's premise
shifted under it: measurement (2026-07-17) showed delegating work always
*raises* total cost — a worker re-pays context loading and verification runs
twice — so delegation buys time-shifting, capacity arbitrage, or true
parallelism, never savings. Practice is now "default inline; dispatch only on
a named dividend," and tier-picking is rare because dispatch is rare. The DoD
as written aims at model-picking; the delivered understanding is an
honest-cost gate. Rescope question routed to the convergence step.

### PO lifecycle

*DoD: the loop market → vision → vision-convergent roadmap → ship runs
repo-local; frontier: discovery + convergence.* The open frontier, actively
converging. This window delivered the full skill pipeline: the route-finding
sequencer was factored into individually callable steps; shaping (turn a
direction into high-level drafts) was split from refinement (make drafts
execution-ready) after a skill audit caught them collapsed; market research
became shallow-by-default with owner-gated depth. The loop has now run once
end-to-end on a real branch (X6). What remains unproven is decision quality at
acceptable ceremony — which is exactly what the current calibration run (this
receipt is its first artifact) is testing. Drift observed and corrected: the
audit step itself rendered prune *verdicts* at audit time, contradicting the
intended order where all decisions land at the final refinement step.

### Introspection & self-improvement

*DoD: every recurring surface, skill, and process can be audited against
reality on evidence, yielding owner-gated verdicts, never ceremony.* The
instruments exist and have drawn blood: a skill audit produced the
shaping/refinement split above; retrospectives and dated receipts are in use.
Two honest gaps. **Propagation:** upgrading the Horus CLI does not refresh
what it previously wrote into projects — 126 stale skill installations across
the fleet; two projects were hand-refreshed 2026-07-20 and the observations
(one command suffices; a zero-pending dry-run is a natural completion receipt;
there is no per-skill selection; nothing drives the commit/PR after writing)
are recorded on the work item that owns the fleet-wide design.
**Self-detection:** the audit-contract drift above was caught by the owner
interrupting, not by any check — the facet audits everything except its own
alignment with owner intent.

### Autonomous dispatch

*DoD: a scheduled worker+supervisor loop runs approved work items end-to-end
under the owner's accounts and a standing pre-authorization.* Built and proven
once. The loop is real: pre-authorizations bind at launch and expire; on-disk
timers survive reboots; the supervisor independently verifies CI on the exact
commit plus a live probe and merges only with explicitly granted authority; a
failure halts dependent scheduled work and lands an aggregated Telegram alert.
A genuine work item was delivered this way during a drill. The bottleneck is
no longer machinery but **fuel**: exactly one work item currently carries the
"ready + eligible for autonomous execution" label, so the loop ran end-to-end
once and then starved. The final step of this calibration chain
(backlog-refinement) is where eligible items get minted — including possibly
the five dead cockpit-era commands above as small, deterministic removal
items. Guard: never manufacture work solely to feed the loop.

### Distribution

*DoD: `uv tool install` yields a safe, current, isolated setup on all three
OSes; the hosted app tracks releases.* Met operationally: five releases in
four days each crossed trusted publishing, package-index verification, fresh
installs on Windows/Linux/macOS, and a hosted redeploy with an origin-side
health check. The public domain (including the health endpoint) now sits
entirely behind an access gate, so verification is origin-side by design. The
one known open rung: updating the hosted app after a release is an instruction
in the project docs, not yet an automated guarantee (already a backlog item).

## 6. Ceremony observations

- The audit staleness reminder fired repeatedly within days because its
  interval counts *releases* (15 in 4 days of dogfooding). Releases are a poor
  clock during rapid iteration — suggest counting releases AND elapsed days.
- The closure-frequency concern deferred by the last audit is closed, not
  re-deferred: the one-boundary rule structurally resolved it (closure commits
  fell from ~50% to ~24% of all commits in the window).
- Native-overlap check: Claude Code's window shipped orchestration-plane
  features (background subagents, agent teams, spawn caps) — a plane Horus
  deliberately cedes; Codex shipped bug fixes. No Horus surface became
  natively covered this window.
