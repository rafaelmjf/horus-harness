# tui-backlog-kanban-board — width-adaptive kanban lens over the backlog

## Why — owner, 2026-07-21

The bold half of the backlog-visualisation idea: a true multi-column **kanban board** in
the TUI, where the same cards lay out as columns generated by a switchable **group-by**
(status / facet / branch / autonomy / readiness). The columns aren't sacred — the group-by
*is* the lens; the board is its wide-width rendering, the grouped list
(`tui-backlog-grouped-list`) is the narrow-width one.

## The make-or-break constraint: geometry

The owner drives Horus **from mobile** (this whole session was mobile). A multi-column
board is hostile to a narrow terminal viewport — thin columns, truncated cards, miserable
horizontal scroll (the pty-geometry failure mode this repo keeps hitting). So the board
MUST be **width-adaptive**: board on wide desktop widths, degrading gracefully to the
grouped list on narrow/mobile. The board never *replaces* the list. Get this wrong and it
is worse than today.

## Depends on

`tui-backlog-grouped-list` — same group-by engine + card set; the board is its wide-width
rendering. Ship the list first (it de-risks and delivers most of the value).

## Open questions

- Width-adaptive strategy: breakpoint down to the list? horizontal scroll with sticky
  columns? column virtualisation?
- How many columns before it's unusable in a terminal; column-collapse behaviour.
- Card-cell content at narrow column widths.
- Does moving a card between columns (e.g. a readiness change) *edit* the card, or stay
  read-only? (See grouped-list non-goal.)

## Non-goals

- Not a list replacement — the list stays the default and the narrow/mobile view.
- Not board-driven card mutation unless geometry + safety are proven first.
