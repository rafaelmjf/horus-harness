# WHAT OPENWIKI IS

OpenWiki is LangChain's open-source agent and CLI for generating and maintaining codebase documentation for coding agents. Its premise is that `AGENTS.md` and `CLAUDE.md` should point to selectively retrieved repository documentation instead of carrying a whole codebase explanation in every prompt. LangChain announced the first release on 1 July 2026 and explicitly described codebases as the initial use case ([launch post](https://www.langchain.com/blog/introducing-openwiki-an-open-source-agent-for-repo-documentation)).

In code mode, a developer installs the npm CLI, runs `openwiki code --init` in a repository, and gets a repository-local `openwiki/` tree of Markdown pages. OpenWiki also creates or updates marker-delimited blocks in both `AGENTS.md` and `CLAUDE.md`, telling agents when to consult that tree. The current README additionally describes a separate personal-memory mode and several non-code connectors, but those are outside this comparison ([README](https://github.com/langchain-ai/openwiki/blob/main/README.md)). The generated output is therefore a wiki-shaped Markdown corpus usable by humans and file-reading agents. The interactive CLI can accept questions, but the public documentation does not establish an exported knowledge graph, stable structured schema, retrieval API, or separately hosted query service for code mode; it is best characterized as docs plus an agent chat, not a queryable capability model.

Freshness is agent-driven and Git-aware rather than automatic on every code edit. `openwiki code --update` supplies changes since the recorded Git head to the documentation agent. The supplied CI examples run daily and open a documentation PR containing `openwiki/` plus the managed instruction files; GitLab and Bitbucket examples are also supplied ([GitHub Actions example](https://github.com/langchain-ai/openwiki/blob/main/examples/openwiki-update.yml), [update implementation](https://github.com/langchain-ai/openwiki/blob/main/src/agent/utils.ts)). A docs-only backend constrains agent writes to the documentation surface, and `.last-update.json` records update provenance; nevertheless, accuracy and completeness still depend on an LLM-generated interpretation of code and diffs ([docs-only backend](https://github.com/langchain-ai/openwiki/blob/main/src/agent/docs-only-backend.ts)).

The software is MIT-licensed and runs locally or in the repository's own CI, using a configured inference provider; optional LangSmith tracing is supported. I found no first-party hosted OpenWiki code-documentation service in LangChain's launch material. The repository and npm package are the distribution/hosting story, so private-code users must assess what source context is sent to their chosen model and tracing providers ([LICENSE](https://github.com/langchain-ai/openwiki/blob/main/LICENSE), [package metadata](https://github.com/langchain-ai/openwiki/blob/main/package.json), [provider configuration](https://github.com/langchain-ai/openwiki/blob/main/README.md#customizing)).

Maturity is the main uncertainty. The public repository was created on 22 June 2026, the announcement followed nine days later, and the latest published release observed for this review was 0.1.0 on 9 July ([repository metadata](https://api.github.com/repos/langchain-ai/openwiki), [releases](https://github.com/langchain-ai/openwiki/releases)). The main branch was already ahead at package version 0.1.1 and the README had changed from the launch post's bare `openwiki --init` interface to explicit `personal` and `code` modes. That is healthy early activity, but also evidence of a fast-moving pre-stable contract. Public material was sufficient to verify architecture and intended workflow, but I could not verify real-world documentation fidelity, update precision over long histories, operating cost on large repositories, private-repository deployments, or independent evaluations.

# COMPARE vs OURS

The overlap is durable, repository-local context that native coding agents discover through their normal instruction files. Both approaches avoid requiring a proprietary always-on memory service, derive useful context from sources already in or around the repository, and aim to reduce repeated whole-repository rediscovery.

They operate at different semantic layers:

- OpenWiki is a broad code-understanding/doc engine. It scans implementation evidence and generates many linked Markdown pages about architecture, workflows, modules, conventions, and source locations. It is better suited to answering “how does this code work?” and can provide substantially richer onboarding and navigation than Horus does today.
- Horus is a compact continuity and fleet-memory layer. `PRD.md` deliberately records human/agent-curated vision, current focus, next action, backlog, shipped capabilities, and load-bearing rules. `capabilities.py` deterministically extracts the Vision lead and Shipped lines into a provenance-stamped JSON catalog; it does not ask an LLM to reinterpret the code. Horus is better suited to “what is this project, what has it shipped, and what should happen next?” across many projects, including decisions and intent that cannot be inferred reliably from source.

OpenWiki would add deep repository explanation, cross-file architectural navigation, and generated onboarding material that Horus intentionally lacks. Its interactive agent also offers a conversational way to explore that corpus. Horus adds several things OpenWiki does not establish: a small machine-readable per-project capability record; fleet-wide aggregation; explicit current work/backlog/session handoff; closure and freshness gates; and a curated distinction between intent, shipped state, and rules. In short, OpenWiki documents implementation shape; Horus preserves project state and exposes a capability index. Neither output safely substitutes for the other.

Integration is conceptually clean only as an optional external capability behind a fit gate: Horus could surface that a project has an `openwiki/` corpus and link agents or the dashboard to it, while continuing to treat `.horus/PRD.md` as continuity truth. Adopting it now would still impose material costs: Node/npm alongside Horus's Python distribution, inference credentials and spend, provider/privacy review, another scheduled PR producer, a large LLM-authored artifact tree to review, and two tools editing `AGENTS.md`/`CLAUDE.md` managed blocks. Marker-delimited blocks reduce direct collision risk, but generated wiki drift and PR noise remain. Importing OpenWiki prose into the capability catalog would also weaken the catalog's deterministic “extract, do not invent” contract. Building an equivalent Horus doc engine would cross the fleet boundary and duplicate a capability the external project already targets.

# RECOMMENDATION

**skip-but-watch.** Do not add OpenWiki as a dependency, ingest its generated prose into `capabilities.json`, or speculatively build a competing Horus documentation engine. Its complementary value is real—deep codebase explanation—but the current 0.1-era interface, unmeasured fidelity, provider/privacy implications, and ongoing generated-doc review cost are not yet justified for Horus's compact memory-plane mission.

Concrete revisit trigger: reopen the fit assessment when OpenWiki publishes a stable 1.x code-mode contract **and** there is public or locally reproducible evidence that incremental updates stay source-grounded and non-stale across at least 30 merged changes in a private, polyglot repository. At that point, run one opt-in pilot whose acceptance gate measures stale/unsupported claims, update PR noise and inference cost; if it passes, spawn a narrowly scoped `optional-openwiki-project-link` card to detect/link an existing `openwiki/` corpus without making OpenWiki required or changing Horus's source-of-truth model.
